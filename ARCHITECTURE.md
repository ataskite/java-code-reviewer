## 💻 环境要求

### 系统要求
- **操作系统**：Linux（支持标准Bash环境）

- ✅ **标准命令** - 仅使用Linux标准命令行工具

### 5. 多维表格字段体系

## 🎯 运行框架特点

### 1. 双Agent架构

```
主Agent (协调者)          子Agent (执行者)
    ↓                          ↓
• 用户交互                  • 代码审查
• 流程控制                  • 问题发现
• 脚本调用                  • 报告生成
• 参数准备                  • 飞书上传
• 结果展示                  • 结果汇总
```

**优势**：
- 主Agent专注用户交互和流程控制
- 子Agent专注代码审查任务
- 长时间审查不阻塞主Agent
- 用户可在审查期间继续使用OpenClaw

### 2. 脚本化工作流

**优势**：
- 脚本可独立维护和测试
- 修改脚本无需更新文档
- 输出格式标准化便于解析
- 可复用于其他工具

### 3. 引用式架构

**优势**：
- 审查框架独立维护
- Agent提示词精简（665行 vs 原862行）
- 版本同步机制清晰
- 修改框架不影响提示词结构

### 4. 变量标准化

**命名规则**：
```bash
# 项目相关
PROJECT_DIR           # 项目根目录
PROJECT_NAME          # 项目名称
PROJECT_TYPE          # Maven/Gradle + 单/多模块
PROJECT_SCAN_RESULT   # 项目扫描结果

# Git相关
CURRENT_BRANCH        # 当前分支
TARGET_BRANCH         # 目标分支
GIT_LOG_OUTPUT        # Git提交记录
CHANGED_FILES_OUTPUT  # 变更文件列表
DIFF_STATS_OUTPUT     # 变更统计

# 审查配置
REVIEW_TYPE           # 审查类型
REVIEW_SCOPE          # 审查范围
REVIEW_MODE           # 审查模式
FEISHU_UPLOAD_OPTION  # 飞书上传选项
```

11. 修复状态、12. 审查模式、13. 审查日期、14. 负责人、15. 备注

**预留修复字段（3个）**：
16. **修复时间** - 日期类型，初始留空
17. **修复分支** - 文本类型，初始留空
18. **修复人** - 人员类型，初始留空

## 🚀 执行流程总结

1. **用户触发** → 主Agent接收请求
2. **阶段一-四** → 脚本采集项目数据
3. **阶段五** → 交互式确认审查配置
4. **阶段六** → 构造prompt并调用子Agent
5. **子Agent执行** → 读取审查框架 + 扫描代码 + 生成报告
6. **飞书上传** → 条件执行（根据用户选择）
7. **结果展示** → 主Agent展示结果给用户

## 🆕 版本 5.1 更新说明

### 多维表格字段扩展（15 → 18）
新增 3 个预留修复字段，用于完整的问题修复追踪：
- **修复时间**（日期）：记录实际修复完成时间
- **修复分支**（文本）：记录修复所在的分支名（如 fix/issue-123）
- **修复人**（人员）：记录实际修复人员
