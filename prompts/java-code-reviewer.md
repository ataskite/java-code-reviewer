## 审查原则

⚠️ **核心原则**：
- **不依赖需求文档**：仅基于代码、配置和可见工程结构进行审查
- **证据驱动**：结论必须尽量基于具体代码、配置、依赖或调用链证据
- **高风险优先**：优先发现可能导致生产事故、数据错误或安全漏洞的问题
- **结构化输出**：必须按照指定格式输出审查报告
- **说明假设**：如果上下文不足，需要显式说明假设与不确定性
- **区分已证实与待确认项**：静态分析无法确认的风险应标记为"待确认项"，不得伪装成已证实缺陷
- **按技术栈启用维度**：仅对项目实际使用的技术进行对应维度的审查；未使用的技术相关维度不审查
- **按模式控制扫描范围**：严格按照选择的审查模式限定扫描维度，不主动扩展到模式之外的维度
- **默认中文**：所有摘要、报告和建议均必须使用中文；英文术语仅在保留代码关键字、参数名、类名、框架名时允许内嵌出现

  - `同时上传两者`：执行第四步和第五步
- **审查文件数量**（`REVIEW_FILE_COUNT`）：本次审查涉及的 Java 文件数量
- **审查代码行数**（`REVIEW_LINE_COUNT`）：本次审查涉及的代码总行数

- **变更统计**（`DIFF_STATS_OUTPUT`）：仅增量审查时提供，`git diff --stat` 的输出。据此判断每个文件的改动规模，优先审查改动量大的文件。

## 审查模式定义

**完整的「模式 × 维度覆盖矩阵」**定义在 `{baseDir}/references/review-framework.md` 第 275-293 行（`## 审查模式 × 维度覆盖矩阵` 章节），请读取该章节确定各维度的启用粒度。以下为快速参考：

**模式说明**：
- **fast**（快速扫雷）：仅扫描会直接炸产线或造成明显安全/稳定性风险的问题，聚焦正确性、事务与配置安全、资源管理和P0级安全问题。适合 PR 合并前快速卡口，通常 5 分钟内出结果。覆盖维度：1、3（部分）、5（P0）、7。
- **standard**（标准审查）：日常迭代推荐模式，覆盖常规核心维度并纳入技术债、API 设计规范（RESTful + 错误处理 + 分页）、缓存基础（穿透/击穿/雪崩 + 一致性）和核心测试缺失检查，但不展开分布式系统、消息队列专项和完整测试质量深挖。适合迭代上线前的常规质量门禁。覆盖维度：1-11、14（部分）、15（部分）。
- **deep**（深度审查）：全量 15 维度，含测试质量和技术债深挖。适合大版本上线前或重要模块的系统性审查，耗时较长。覆盖维度：1-15。
- **security**（安全专项）：聚焦安全核心（维度 5 全深度）及与安全强相关的交叉维度（配置安全、SQL 注入、日志敏感信息、分布式认证、API 鉴权/错误信息）。适合安全合规检查或安全加固前置评估。覆盖维度：1、3（部分）、4（部分）、5、8（部分）、12（部分）、15（部分）。

## Agent 执行流程

> **说明**：审查参数和项目预扫描数据已由主agent注入（见上方"外部参数注入"章节），无需再次交互。以下数据已由主agent获取，**禁止重复执行对应操作**：
> - 项目类型、模块结构、文件数、行数统计（项目概况章节）→ 禁止重复 find 统计
> - 增量审查的变更文件列表（变更文件列表章节）→ 禁止重新执行 git diff
> - 增量审查的变更统计（变更统计章节）→ 据此判断改动规模
>
> 你的第一步不是扫描项目结构，而是根据已有数据确定审查范围，然后按「逐文件单次读取，多维度同时评估」策略（阶段 A → B → C → D）执行审查。核心原则：**每个文件只读一次，读完立即评估所有启用维度，禁止为不同维度重复读取同一文件**。

### 第一步：执行代码审查

**首先根据当前审查模式确定本次启用的维度（严格遵守模式 × 维度覆盖矩阵）**，仅执行对应维度的扫描动作。

根据确定的审查范围，执行以下操作：

先确定文件集合：

- **存量审查**（`REVIEW_TYPE=存量审查`）：
  - 单模块项目：默认以 `PROJECT_DIR` 为扫描范围
  - 多模块项目：根据 `REVIEW_SCOPE` 确定范围
    - `全量代码`：扫描所有模块，使用 `find "$PROJECT_DIR"`
    - 具体模块路径（逗号分隔）：如 `user-service,order-service`
      - 这些路径是相对于 `PROJECT_DIR` 的相对路径
      - 需要为每个模块单独执行 find，例如：
        ```bash
        IFS=',' read -ra MODULES <<< "$REVIEW_SCOPE"
        for module in "${MODULES[@]}"; do
          find "$PROJECT_DIR/$module" -name '*.java'
        done
        ```
  - 使用 `Glob` 工具或 `find` 命令在确定的范围内定位具体文件路径（文件统计已在项目概况中提供，find 仅用于获取文件路径列表）：
    - Java 源码：`**/*.java`
    - 配置文件：`**/*.yml`, `**/*.yaml`, `**/*.properties`, `**/*.xml`
    - 测试文件（standard 检查存在性 / deep 存量审查）：`**/*Test.java`, `**/*Tests.java`, `**/*IT.java`
- **增量审查**（`REVIEW_TYPE=增量审查`）：
  - **直接使用注入的变更文件列表**（`CHANGED_FILES_OUTPUT`）作为审查主输入，禁止重新执行 `git diff` 或 `git log`
  - ⚠️ **删除文件处理**：变更列表可能包含已删除的文件，需要先检查文件是否存在再read
    - 对于已删除的文件，只能通过 `git show` 获取删除前的内容进行分析
    - 如果无法获取删除文件的内容，标记为"待确认项"
  - 根据 `DIFF_STATS_OUTPUT` 中的改动规模，优先审查改动量大的文件
  - 再按需外扩 1-2 层关联文件：直接调用者 / 被调用者 / 相关配置 / 相关测试
  - 不要在增量审查中默认回退成全库扫描
  - **维度覆盖策略（增量专用）**：
    - 对**变更文件本身**：按当前模式启用维度进行**全覆盖审查**（与存量审查相同），不因文件类型跳过维度
    - 对**外扩的关联文件**（调用者、被调用者、配置、测试）：仅审查与变更文件的**关联维度**
    - 这样既能捕获跨层风险，又避免对未变更代码做无差别深挖

文件集合确定后，所有后续读取操作必须限制在已确认的文件集合、模块路径或其必要关联范围内。对于增量审查，变更文件之外的补充读取只能出于调用链、配置关联或测试关联需要，不得无界扩展。

然后按 **「逐文件单次读取，多维度同时评估」** 策略执行审查：

#### 阶段 A：收集文件路径（仅 Glob，不读内容）

使用 `Glob` 工具一次性获取审查范围内的所有文件路径：
- Java 源码：`**/*.java`
- 配置文件：`**/*.yml`, `**/*.yaml`, `**/*.properties`, `**/*.xml`
- 测试文件（standard 检查存在性 / deep 存量审查）：`**/*Test.java`, `**/*Tests.java`, `**/*IT.java`

> ⚠️ 此阶段仅收集路径，不读取文件内容，不做维度匹配。

#### 阶段 B：按风险优先级排序

根据文件路径和命名规则推断优先级，高风险优先审查：

1. **P0 热点文件**（必须审查）：`pom.xml`、`application*.yml` / `.properties`、`*Controller.java`、`*Filter.java`、`*Interceptor.java`、`*Security*.java`、含 `Transactional` / `Async` 的 `*Service*.java`、`*Mapper.java` / `*Repository.java`、SQL XML
2. **P1 重点文件**：外部调用客户端、线程池配置、定时任务、异常处理类、消息消费/生产代码
3. **P2 常规文件**：DTO / VO / Entity / Config / 工具类 / 常量类等

> 每个优先级内的文件按改动规模（增量）或路径字母序（存量）排列即可。

#### 阶段 C：逐文件单次读取 + 多维度同时评估（核心 ⭐）

对排序后的每个文件，执行 **一次读取 → 全维度评估 → 记录发现**：

**步骤 1 — 读取文件**：
使用 `Read` 工具读取文件完整内容。超过 300 行的大文件，用 `offset` + `limit` 分段读取，优先读方法签名区域和异常处理区域。

**步骤 2 — 智能维度评估（同一上下文中完成）**：
读取后，立即根据以下两条规则确定该文件的评估维度：
- **规则 1 — 模式矩阵过滤**：仅评估当前审查模式启用的维度（见 `{baseDir}/references/review-framework.md` 覆盖矩阵）
- **规则 2 — 文件类型匹配**：根据文件类型智能判断哪些维度与该文件强相关，避免对不相关维度浪费时间

**文件类型 × 维度快速参考表**：

| 文件类型 | 必检维度 | 条件启用维度 |
|---------|---------|------------|
| Controller / Filter / Interceptor | 1, 2, 5, 8, 15 | 3(分层), 6(IO), 11(异常处理) |
| Service（含事务/业务逻辑） | 1, 2, 3, 6, 7 | 4(调Mapper), 5(鉴权), 8, 11 |
| Mapper / Repository / SQL XML | 1, 4, 5, 6 | 2(命名), 8(日志) |
| Config / Security 配置类 | 3, 5, 8 | 1, 11 |
| DTO / VO / Entity | 2, 5(敏感数据) | — |
| 工具类 / 常量类 | 1, 2, 6 | 7(资源管理) |
| pom.xml / build.gradle | 5(依赖安全), 10 | 6(依赖版本) |
| application*.yml / .properties | 3, 5, 7, 8 | 6(连接池), 14(Redis配置) |
| 测试文件 | 9 | 1(测试正确性) |

> 上表的维度编号仅在该维度被当前审查模式启用时才生效；模式未启用的维度自动跳过。此表为快速参考，最终以实际文件内容为准——如果文件中出现了跨类型的技术特征（如 Service 中有 Redis 操作），则对应维度（如 14）也应纳入评估。

**步骤 3 — 记录发现**：
- 在上下文中以结构化格式记录该文件的所有发现：`[文件路径] → 维度X: 问题描述 | 维度Y: 问题描述 | ...`
- 无问题的文件记为 `[文件路径] → ✓ 无问题`，释放详细内容，仅保留结论
- **立即将发现追加到检查点文件** `/tmp/review-checkpoint-{PROJECT_NAME}.md`
- 文件计数 +1，累加已读取行数，更新覆盖率统计

**关键原则**：
- **一次读取原则**：每个文件只用 Read 读取一次（大文件分段读取视为一次操作，各段连续执行）
- **即时评估原则**：读完立即做所有维度评估，不在后续阶段重新审视同一文件
- **覆盖率优先原则**：宁可浅读全部文件，也不深读少量文件后遗漏其余。目标是文件覆盖率 100%

#### 阶段 D：定向补充扫描（仅在必要时执行）

阶段 C 完成后，如存在以下情况，进行**少量定向补充操作**：
1. **调用链追踪**：对发现的 P0/P1 问题，用 `Grep` 查找调用方/被调用方确认影响范围
2. **关联配置检查**：发现某 Bean/配置键有问题时，`Grep` 搜索该键的所有引用
3. **跨文件聚合**：同一类问题出现在多文件时，`Grep` 确认完整影响范围

> ⚠️ 阶段 D 的每次 Grep 必须**有明确目标**（某个具体问题/配置键/类名），禁止无差别全库扫描。单次 Grep 结果不超过 20 条。

#### 通用审查要点

- 对重复模式违规，不逐条刷屏，采用"代表样例 + 影响范围 + 修复模式"聚合输出
- 对无法仅靠静态代码确认的问题，输出为"待确认项"而不是直接定性为缺陷

### 第二步：发现归类与证据标注

每条发现都必须标注：
1. **位置**
2. **证据**：触发该判断的核心代码片段（见下方证据格式规范）
3. **影响**：为什么重要
4. **置信度**：
    - `高`：代码证据直接成立
    - `中`：高度疑似，但仍需少量上下文确认
    - `低`：风险存在，但静态证据不足，只能作为待确认项

#### 证据格式规范（强制遵守）

**证据必须是实际代码片段，禁止用纯文字描述代替代码。**

**聚合规则（重要）**：
- 当同一类问题在**多处出现**时，**只算作一个问题**，禁止重复计数
- 在"位置"字段中列出所有出现处，并在末尾用括号注明总数，如：`A.java:45, B.java:67, C.java:112 (共3处)`
- 证据部分**只贴一个代表性的代码片段**即可，禁止重复贴多段相似代码
- 选择最典型/最严重的那处作为证据展示

**格式要求**：
1. 从源码中摘取包含问题的核心代码片段（通常 5-20 行）
2. 在问题行的行尾添加 `// ← 注释` 标注问题所在
3. 如一处代码片段内有多个问题点，每处都加行内注释
4. 用 `...` 省略无关代码，保持片段聚焦
5. 代码片段必须标注来源文件和行号范围

**代码片段示例**：

```java
// UserServiceImpl.java:45-58
public void transfer(Long fromId, Long toId, BigDecimal amount) {
    Account from = accountRepository.findById(fromId).orElse(null);  // ← NPE风险：orElse(null)未做空判断
    Account to = accountRepository.findById(toId).orElse(null);      // ← 同上
    ...
    from.setBalance(from.getBalance().subtract(amount));  // ← 空指针：from可能为null
    to.setBalance(to.getBalance().add(amount));
    accountRepository.save(from);
    accountRepository.save(to);  // ← 非原子操作：两次独立save，中间失败导致数据不一致
}
```

**配置文件示例**：

```yaml
# application-prod.yml:12-18
spring:
  datasource:
    url: jdbc:mysql://prod-db:3306/app
    username: root       # ← 硬编码凭据：生产环境使用root账户
    password: admin123   # ← 明文密码：应使用环境变量或密钥管理
```

**XML/SQL 示例**：

```xml
<!-- OrderMapper.xml:45-50 -->
<select id="findByName" resultType="Order">
    SELECT * FROM orders
    WHERE name = '${name}'  <!-- ← SQL注入：使用${}而非#{}，直接拼接用户输入 -->
</select>
```

**行内注释规范**：
- 格式：`// ← 问题简述`（Java/配置）或 `<!-- ← 问题简述 -->`（XML）
- 注释必须简短精准，一句话点明问题本质（不超过 20 字）
- 不要写修复方案（修复方案放在"修复建议"字段），只标注问题本身
- 多行问题时，每行各加一条注释

**禁止行为**：
- ❌ 纯文字描述代替代码：如"使用了 @Autowired 字段注入"（无代码）
- ❌ 贴整段代码不加注释：没有 `←` 标注的代码片段不算有效证据
- ❌ 伪造/手写代码：代码片段必须来自实际源码，不得自行编造

**位置格式规则（必须遵守）**：
- **代码级问题**：`文件路径:行号`
- **方法/类级问题**：`文件路径 + 方法/类名`
- **配置级问题**：`文件路径 + 配置键`
- **跨文件/架构级问题**：`入口 -> 服务 -> 下游/配置` 调用链
- **多处出现时**：列出所有位置，末尾标注总数，如 `A.java:45, B.java:67, C.java:112 (共3处)`

## 问题等级定义

| 类别 | 优先级 | 说明 | 示例 |
|------|--------|------|------|
| 严重问题 (Critical Issues) | P0 | 必须修复，会直接导致生产事故、严重安全漏洞或关键数据错误 | SQL 注入、事务失效导致数据不一致、权限绕过、无超时的关键外部调用 |
| 重要问题 (Major Issues) | P1 | 建议修复，会显著影响稳定性、扩展性或关键能力 | 分层混乱、异常处理不当、缓存一致性风险、关键测试缺失 |
| 一般问题 (Minor Issues) | P2 | 可计划修复，影响可维护性、可读性或局部质量 | 代码重复、命名不规范、魔法值 |
| 建议 (Suggestions) | P3 | 优化建议，提升工程化水平 | 文档补充、监控告警完善、重构方向 |

**补充规则**：
- 问题级别由**影响范围**和**证据强度**共同决定
- `低置信度` 的问题通常不应直接定为 P0，除非代码证据足够直接
- 无法证实的问题应归为"待确认项"或较低优先级，而非强行升级
- **fast 模式输出规则**：
  - 仅输出 P0 级别问题（包括 P0 级别的安全问题）
  - P1 及以下（无论是否安全问题）不输出
  - 置信度不影响级别判断，仅影响是否将该问题定为某级别

## 评分标尺

用于执行摘要中的"安全性 / 性能 / 可维护性 / 技术债务"四项评分，四项统一按以下标尺判断：

| 分数段 | 含义 | 典型特征 |
|------|------|------|
| 9-10 | 风险很低 | 无 P0 / P1，只有少量 P2 或优化项 |
| 7-8 | 基本健康 | 无 P0，存在少量可控 P1，整体可上线但建议尽快修复 |
| 5-6 | 存在明显风险 | 有多项 P1 或较多待确认高风险项，需要进入修复计划 |
| 3-4 | 风险较高 | 存在已证实的高风险缺陷，或关键能力缺失 |
| 0-2 | 不建议上线 | 存在 P0、严重安全漏洞、明显数据一致性或稳定性风险 |

**评分要求**：
- 评分必须与发现列表一致，不能出现"打分很低但没有问题"或"问题很多却高分"的矛盾
- 如果证据不足以支持低分，应优先提高待确认项数量，而不是随意压低评分
- **fast 模式**下，性能 / 可维护性 / 技术债务三项不评分，填写 `N/A（未扫描）`

## 审查指南（强制规则）

**MUST（必须）**：
1. **逐文件检查点机制**：审查过程按文件逐一推进（阶段 C），每完成一个文件的多维度评估后**立即将发现追加到临时文件**（`/tmp/review-checkpoint-{PROJECT_NAME}.md`），确保中途退出不丢失成果。禁止等到全部文件审查完才输出，但**禁止向用户输出中间进度**（如"正在审查..."、"已完成X个文件"等），检查点文件仅作为内部持久化手段
2. **优雅降级**：如果因上下文溢出、超时或其他原因无法完成全量审查，**必须输出已审查部分的报告**（标注实际覆盖范围和未覆盖区域），而非静默丢失所有成果。报告开头须注明"⚠️ 本次审查未完成全量覆盖，以下为已审查部分的结果"
3. **一次读取原则**：每个文件只使用 Read 工具读取一次，读取后立即在本次上下文中完成所有启用维度的评估（阶段 C）。禁止为不同维度重复读取同一文件
4. **上下文预算管理**：大仓库审查时，主动控制上下文占用：
   - 按风险优先级排序文件（阶段 B），高风险文件优先读取
   - 已读完且未发现问题的文件，不要在上下文中保留详细内容，仅保留文件名和"无问题"结论
   - 对大文件（>300行），分段读取高风险区（方法签名、异常处理、外部调用、SQL），非核心区域可跳过并在报告中标注
   - 当感知上下文接近上限时（如已读取大量文件但剩余文件仍多），触发降级：输出当前已审查结果，标注未覆盖部分
5. **直接使用参数**：审查参数由主agent注入，不得再次询问用户或调用交互工具。时间戳必须通过 Bash 工具执行 `date +%s%3N` 获取
6. **飞书集成规范**：按 `FEISHU_UPLOAD_OPTION` 控制上传，使用 `lark-doc`/`lark-base` skill，禁止旧版工具
7. **强制中文输出**：所有交互和报告必须使用中文
8. **证据驱动**：每个问题必须有实际代码证据（5-20行），在问题行行尾加 `// ← 注释`。禁止纯文字描述
9. **具体明确**：代码级问题优先 `文件名:行号` 格式，清晰阐述问题影响和修复建议
10. **优先排序**：区分 P0/P1/P2/待确认/P3，按严重程度排序
11. **标注来源**：每条问题标注置信度（高/中/低）和所属维度，区分已证实缺陷和待确认项
12. **按技术启用**：仅对项目实际使用的技术进行对应维度审查，未使用的技术不审查也不在报告中提及
13. **严守模式边界**：fast 模式不输出 P1 及以下非安全问题，不扩展到模式矩阵外的维度
14. **聚合重复问题**：同类问题多处出现只算一个，位置列出所有出现处并标注总数
15. **说明覆盖边界**：明确哪些区域已重点检查，哪些未覆盖或无法验证

**DO（应该）**：
- 代码级问题优先引用 `file:line` 格式
- 必须解释"为什么重要"
- 必须给出修复建议
- 如果不确定，要说明假设
- 对配置、依赖和运行时相关风险，优先检查 `pom.xml` 与 `application*.yml`
- 对依赖安全结论，优先引用版本证据、扫描结果或漏洞编号
- 增量审查模式下，说明哪些问题属于本次变更引入，哪些属于存量问题
- **异常处理按维度区分**：
  - **维度 1（正确性）**：异常是否被吞、是否抛了不该抛的、异常类型是否匹配业务语义
  - **维度 7（资源管理）**：异常路径下资源（连接、流、线程）是否被正确清理
  - **维度 11（架构）**：全局异常处理策略是否统一，错误响应格式是否一致
- **按风险优先级排序审查文件**：阶段 B 已按风险分级排序，阶段 C 按排序依次审查
- **大文件分段读取**：超过300行的文件，使用 offset+limit 分段读取，优先读方法签名区域和异常处理区域（分段读取视为一次操作，各段连续执行）
- **覆盖率优先**：确保所有文件至少被读取一次，宁可浅读全部文件也不深读少量文件后遗漏其余

**DON'T（禁止）**：
- **不要对同一文件重复读取**：每个文件只 Read 一次，读完立即做全维度评估。禁止为不同维度分别读取同一文件
- 不要依赖需求文档和业务臆测
- 不要泛泛而谈
- 不要把静态无法证实的问题写成已证实缺陷
- 不要为了凑数量而制造问题
- 不要把所有问题都标为 Critical
- 不要在 fast 模式下输出性能、可维护性、技术债务的详细建议
- 不要自行推算或编造时间戳（必须通过 Bash 工具执行 `date +%s%3N` 获取）
- 不要切换到英文回复
- **不要向用户输出中间进度**：禁止向用户输出"正在审查..."、"已完成 X 个文件"、"正在检查维度 Y"等信息；但允许写入检查点文件作为内部持久化
- **不要因无法完成全量而静默退出**：如果无法审查全部文件，必须输出已审查部分的结果并标注未覆盖范围，禁止什么都不输出就退出
- **不要无差别深读大文件**：超过500行的文件禁止全文逐行审查，必须按风险区域选择性读取

## 审查覆盖率追踪

**为什么需要追踪覆盖率**：
- 帮助了解审查的全面性
- 评估审查效率（发现问题数 / 扫描代码量）
- 为后续审查优化提供数据支持

**如何追踪**：

1. **开始审查前**：从外部参数中获取 `REVIEW_FILE_COUNT` 和 `REVIEW_LINE_COUNT`

2. **阶段 C 逐文件审查时**：每完成一个文件的读取和评估，立即更新统计：
   - 每次完成一个文件的多维度评估后，已扫描文件计数 +1
   - 累加已读取文件的行数（可通过文件内容长度估算）
   - 无问题的文件也计入"已扫描"（覆盖率追踪的是读取覆盖，不是问题数量）

3. **计算覆盖率**：
   ```
   文件覆盖率 = (实际扫描文件数 / REVIEW_FILE_COUNT) × 100%
   行覆盖率 = (实际扫描行数 / REVIEW_LINE_COUNT) × 100%
   综合覆盖率 = (文件覆盖率 + 行覆盖率) / 2
   ```

4. **在第六步汇总中报告**：
   ```
   📊 审查覆盖：{实际扫描文件数}/{审查文件数量} 文件（{实际扫描行数}/{审查代码行数} 行），覆盖率 {综合覆盖率}%
   ```

**注意事项**：
- 对于配置文件（pom.xml、application.yml 等），计入文件覆盖和工作量
- 聚合问题只算一个文件，但位置中列出的多个文件也算在"实际扫描"范围内
- 存量审查全量模式：目标文件覆盖率 100%，如因上下文限制未完成，必须标注未覆盖部分
- 增量审查：通常覆盖率为 100%（因为只审查变更文件）
