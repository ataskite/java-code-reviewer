# 审查输出格式规范

本文件定义代码审查报告的输出格式，供子 agent 在第三步（生成审查报告）时参考。

**语言要求**：整份报告必须使用中文撰写；代码标识符、配置键、类名、方法名、框架名可保留原文。

**核心排列规则**：所有问题严格按 **P0 → P1 → P2 → P3 → 待确认** 优先级排序输出。**不按维度分组**，每条问题通过「所属维度」字段标注其所属审查维度即可。

---

### ⚙️ 审查配置快照（报告头部必须展示）

> 本节记录外部注入的所有审查配置参数，确保报告结论与审查边界可追溯。

必须包含：
- 生成时间
- 审查类型
- 审查范围
- 审查模式
- 启用维度
- 跳过维度
- 模式说明
- 检测到的技术栈
- 文件覆盖率

### 📊 审查范围说明
- **项目类型**
- **审查范围**
- **审查时间**
- **审查基准路径**
- **检测到的技术栈**
- **启用的审查维度**
- **跳过的审查维度**
- **覆盖方式**

### 📊 执行摘要

必须包含：
- 整体结论
- 最高风险点
- 安全性 / 性能 / 可维护性 / 技术债务评分
- 问题汇总表
- 总计问题数

### 🔍 覆盖限制与未审查范围

必须明确：
- 已重点检查
- 未充分覆盖
- 限制说明

### ✅ 最佳实践亮点

可写亮点；如无明显亮点，可明确写未发现足够稳定且值得单独表扬的最佳实践亮点。

### 🔴 严重问题 (Critical Issues)

每条格式（严格按此顺序输出）：

---
**问题编号**：P0-N

**位置**：文件路径:行号（同类问题多处出现时，只列一个代表性位置并标注总数，如 `A.java:45（同类问题共3处）`，不逐个列出所有位置）

**置信度**：高/中/低 | **所属维度**：维度名称

**问题**：一句话描述

**证据**：
```
（仅贴一个代表性代码片段，含 `// ←` 行内注释标注问题行，格式见"第二步：证据格式规范"）
```

**影响**：为什么重要，可能造成的后果

**建议**：修复方案，如涉及代码改动可附修复后的代码片段
